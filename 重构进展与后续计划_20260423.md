## 四、当前剩余风险与边界

### 1. 文件预览能力依赖后端补齐

识别复核页左栏目前仍然没有真实文件内嵌预览 URL，因此只能优雅降级到元信息与占位展示。

### 2. 自动化回归还未形成

当前已经完成人工浏览器级 smoke 验证，但还没有把新的工作台链路沉淀成稳定 E2E 脚本。

### 3. 人员身份仍是前端占位

部分复核动作中的 `reviewed_by` 仍然是前端固定占位值，后续如果要进入真实协作场景，需要接入真实身份来源。

## 五、后续计划（按优先级）

### P0：补 E2E 自动化回归

优先级最高。原因是当前最值钱的不是继续堆页面，而是防止已有工作台被后续迭代打回原形。

建议覆盖：

1. 创建项目
2. 上传证据
3. OCR
4. 字段抽取
5. 字段复核
6. 记录生成
7. 记录审批
8. 项目导出

### P1：补页面级真实操作回归

目标是把目前的“页面渲染 smoke”升级为“页面操作 smoke”，重点验证：

- 按钮禁用条件
- 表单提交流程
- 状态标签流转
- 导出门槛判断
- 设备筛选映射

### P2：补后端证据预览/下载接口

目标是给识别复核页左栏提供真实文件能力，而不是继续停留在占位态。

### P3：补质量闸门与 CI

建议逐步固定以下闸门：

- 前端 build
- 后端 health / API smoke
- 浏览器关键路由 smoke
- E2E 回归

## 六、建议执行顺序

建议顺序：

1. E2E 自动化回归
2. 页面操作 smoke 收口
3. 后端文件预览能力
4. 持续校验与 CI

原因很明确：

- 当前工作的最大风险已经不是“页面有没有”，而是“后续改动会不会把现有工作台打坏”
- 所以第一抓手不是继续加需求，而是先把闭环验证工具化

## 七、阶段复盘

### 回顾目标

目标是把前端从功能页提升为统一工作台，并保持后端 API 不变。

### 评估结果

目标已经达成主体部分，关键页面、组件和路由信息架构均已重构完成，并通过构建与浏览器渲染 smoke 验证。

### 分析原因

本轮能推进较快，是因为没有在页面里重复造轮子，而是先抽工作台壳层、状态组件和统计卡片，再逐页落地。

### 沉淀规律

后续同类重构必须优先抽公共抓手，再谈页面收口；否则页面越多，维护成本越高，风格和状态语义就越容易失控。
