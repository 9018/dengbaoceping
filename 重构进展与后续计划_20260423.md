## 二、本轮实操中补掉的两个关键交付缺口

### 1. 导出下载地址不应写死开发机地址

实操发现前端导出下载 URL 仍然写死到 `127.0.0.1:8000`，会导致 Docker 部署下浏览器下载失败。

已修正：

- `frontend/src/api/http.ts`
- `frontend/src/api/exports.ts`

当前策略：统一复用 `apiBaseUrl` 生成导出下载地址。

### 2. 前端容器构建不应继承本地 Vite 开发地址

实操发现前端静态构建会把本地开发 API 地址打进容器产物，导致页面在 Docker 部署时出现 `Network Error`。

已修正：

- `frontend.Dockerfile` 默认注入 `VITE_API_BASE_URL=/api/v1`

当前策略：

- 本地 Vite：通过 `frontend/.env.example` 指向本机后端
- Docker/Nginx：默认走相对地址 `/api/v1`

## 四、后续计划（按优先级）

### P0：补前端 E2E 自动化

#### 目标

把本轮人工浏览器黄金路径沉淀为可重复执行的自动化校验，避免后续改动再次破坏交付链路。

#### 建议范围

- 固化“创建项目 -> 导出下载”的前端 E2E 用例
- 至少覆盖：创建、上传、OCR、抽取、复核、审批、导出
- 将其与现有后端测试区分开，作为交付回归层

#### 验收标准

- 一条命令可跑完整前端黄金路径
- 能发现 API baseURL、下载地址、按钮状态等前端交付问题

### P1：补查询/筛选/审计展示能力

#### 目标

把当前“能做”提升到“更好查、更好复核、更好追溯”。

#### 建议范围

- 字段列表筛选
- 记录状态筛选
- 导出任务列表可读性增强
- 审计日志展示或查询接口增强

#### 验收标准

- 复核人员可以快速定位待处理对象
- 导出历史和复核痕迹可直接查询

### P2：推进真实 OCR provider 接入

#### 目标

在不破坏当前 mock 闭环的前提下，完成真实 OCR 的最小接入闭环。

#### 建议范围

- 配置 endpoint / key / timeout
- 实现 `real_adapter`
- 增加 provider 级测试
- 明确真实 OCR 返回与 `ocr_result_json` 的映射

#### 验收标准

- `OCR_PROVIDER=real` 时可完成真实调用
- 失败有明确错误，成功结果可继续流入字段抽取链路

### P3：补持续校验抓手

#### 目标

把当前人工执行的验证步骤收口成固定质量闸门。

#### 建议范围

- 后端 `pytest`
- 前端 `npm run build`
- 如条件允许，再补 Compose smoke check 或 E2E 触发脚本

## 五、推荐执行顺序

建议按下面顺序推进：

1. 前端 E2E 自动化
2. 查询 / 筛选 / 审计展示
3. 真实 OCR 接入
4. CI / 质量闸门固化

原因：

- 当前最值钱的资产已经不是“再多一个功能”，而是“保证现有整条交付链路不回退”
- E2E 自动化可以直接把这次人工实操沉淀成长期抓手
- 查询与审计属于提升使用效率
- 真实 OCR 接入属于能力增强，但不应先于稳定性收口
