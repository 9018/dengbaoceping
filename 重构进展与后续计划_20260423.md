# 重构进展与后续计划

> 更新时间：2026-04-23
> 目标：将旧 Go 单体上传页 / 静态 HTML / ES 驱动链路，重构为 Python 3.11 + FastAPI + SQLAlchemy + Alembic 的前后端分离业务工具。

---

## 一、当前已经完成的内容

### 1. 重构方向与总体方案已明确
已完成以下设计工作：
- 明确旧项目的主要问题：
  - Go 单体上传页 + 静态 HTML 单页，不适合长期演进
  - 业务状态更多沉淀在脚本、JSON、目录中，而不是稳定的数据模型中
  - 当前依赖 ES/Kibana/Redis/Minio/Postgres 的部署方式偏重，不符合新目标
- 明确新目标技术栈：
  - 后端：Python 3.11 + FastAPI
  - 前端：Vue 3 + Element Plus
  - 数据库：SQLite（开发）/ PostgreSQL（生产）
  - 文件存储：本地 uploads / exports / snapshots
- 明确核心业务对象：
  - Project
  - Asset
  - Evidence
  - ExtractedField
  - EvaluationItem
  - Template
  - EvaluationRecord
- 明确业务主流程：
  - 项目 -> 设备 -> 上传证据 -> OCR -> 字段抽取 -> 条目匹配 -> 模板生成 -> 人工复核 -> 导出结果

## 四、建议的后续实施顺序

### Phase 1：接管文件入口
优先实现：
1. 文件上传 API
2. 本地文件保存
3. Asset 自动创建
4. Evidence 自动初始化

原因：
- 这是整个业务工具的真正入口
- 不先接管文件入口，后面的 OCR / 抽取 / 记录生成都只能停留在静态数据层

### Phase 2：打通 OCR 与抽取闭环
优先实现：
1. OCR mock API
2. OCR 适配接口抽象
3. 字段抽取服务
4. ExtractedField CRUD / review API

原因：
- 先把“证据 -> 抽取字段”这一段跑通
- 这样系统才开始具备可复核能力

### Phase 3：初始化规则和模板
优先实现：
1. 规则 JSON 结构设计
2. evaluation items 初始化导入
3. templates 初始化导入
4. rule loader 升级为真正的规则服务

原因：
- 条目匹配与记录生成依赖规则底座
- 这一步是把“文件化规则”真正变成系统能力

### Phase 4：记录生成与人工复核
优先实现：
1. 记录生成 API
2. EvaluationRecord CRUD / review API
3. 记录与证据关联
4. 人工复核状态流转

原因：
- 这是业务工具区别于 demo 的关键点
- 系统要从“能存数据”升级成“能产出记录”

### Phase 5：项目导出
优先实现：
1. 导出服务
2. 导出 API
3. 导出落盘
4. 导出记录追踪

原因：
- 导出是最终业务交付结果
- 没有导出，这个工具链就还没闭环

### Phase 6：前端最小可用版本
优先实现页面：
1. 项目列表页
2. 项目详情页
3. 设备管理页
4. 证据管理页
5. 识别/复核页
6. 记录中心页
