- 在 `factory.py` 中补齐 `paddle` provider 分发
- 统一 OCR 结果结构，稳定返回：
  - `provider`
  - `status`
  - `full_text`
  - `lines`
  - `processed_at`

- `full_text` 统一由 `lines` 派生

- Paddle 引擎改为 lazy init / 单例缓存，避免每次请求重复初始化
- `mock_adapter` 同步补齐 `lines`，保证默认开发态和真实 provider 输出语义对齐

部分复核动作中的 `reviewed_by` 仍然是前端固定占位值，后续如果要进入真实协作场景，需要接入真实身份来源。

## 七、阶段复盘

### 回顾目标

目标是把前端从功能页提升为统一工作台，并保持后端 API 不变。

### 评估结果

目标已经达成两块主体成果：

1. 前端工作台化升级完成，关键页面、组件和路由信息架构均已重构完成，并通过构建与浏览器渲染 smoke 验证。
2. 后端 PaddleOCR provider 已完成首轮接入，并通过 OCR 定向测试与后端全量回归验证。

### 分析原因

本轮能推进较快，是因为没有在页面里重复造轮子，而是先抽工作台壳层、状态组件和统计卡片，再逐页落地；在 OCR 接入阶段也坚持以既有 adapter/factory 契约为边界，没有去重写 service 消费层。

### 沉淀规律

后续同类重构必须优先抽公共抓手，再谈页面收口；同类 provider 接入必须优先守住契约边界，再补配置、测试和文档闭环，否则改动面会迅速失控。
