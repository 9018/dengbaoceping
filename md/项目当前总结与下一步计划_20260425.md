# 项目当前总结与下一步计划（2026-04-25）

## 一、项目当前状态总结

当前项目已经形成“项目管理 → 证据上传 → OCR → 字段抽取/复核 → 测评记录生成/审批 → 项目导出”的主链路闭环，并在此基础上新增了两条知识化辅助能力：指导书知识库、历史人工测评记录库，以及两者之间的规则关联能力。

### 1. 已完成的核心能力

#### 业务主链路
- 项目创建与管理
- 证据上传与归档
- OCR 识别（mock / paddle / real 占位）
- 字段抽取与人工复核
- 测评记录生成、复核与审批
- 项目级 TXT 导出

#### 指导书知识库
- 固定读取 `md/指导书.md`
- 支持章节导入、列表查询、关键词搜索、章节详情查看
- 支持提取 `keywords_json`、`check_points_json`、`evidence_requirements_json`、`record_suggestion`

#### 历史人工测评记录库
- 支持 `.xlsx` 导入历史人工三级测评记录
- 支持筛选、关键词搜索、相似记录查询、句式统计、详情查看
- 支持按 sheet / 控制点 / 符合情况 / 资产类型进行检索

#### 指导书 ↔ 历史人工记录关联能力
- 新增独立关联层 `guidance_history_links`
- 支持按指导书章节重算历史样本关联并持久化
- 支持查询某个 guidance 条目对应的相似历史记录
- 支持从 history 记录反查关联到的 guidance 章节
- 前端指导书详情抽屉已支持：相似历史记录列表、符合情况筛选、刷新关联

## 二、当前工作重点判断

当前仓库已经不再是“只做主链路”的状态，而是进入“主链路 + 知识库 + 历史样本 + 样本关联”的增强阶段。底层逻辑上，现阶段的核心目标不是继续堆功能，而是把已有能力打磨成稳定可交付版本，为后续“基于 guidance 与历史样本增强记录生成”预留抓手。

## 三、下一步计划

### P0：交付闭环
1. 完成浏览器端实操回归：
   - 指导书导入
   - guidance 详情查看
   - 刷新关联
   - 相似历史记录筛选
   - 历史记录库页面基础操作回归
2. 检查 Docker 交付链路是否与新增能力保持一致
3. 确认 README、页面入口、路由、API 文档描述一致

### P1：稳定性与体验增强
1. 优化 guidance-history 关联得分解释，让命中原因更可读
2. 增加空数据提示与边界提示的一致性
3. 评估 guidance-history link 重算接口是否需要批量入口或后台任务化

### P2：为记录生成增强做准备
1. 设计 guidance 与 evaluation item 的显式映射方案
2. 在记录生成前增加“关联 guidance / 历史样本参考”读取能力
3. 评估是否需要把高分历史样本作为记录生成辅助输入

## 四、已识别风险与注意事项

1. 当前关联能力仍是规则匹配第一版，未引入向量检索或语义模型，匹配效果依赖关键词、文本重合和资产类型推断。
2. guidance-history link 当前采用“按 guidance 章节重算并覆盖”的策略，适合现阶段验证，但后续若数据量扩大，需要关注性能与异步化。
3. 当前功能虽然已通过测试和构建，但浏览器端完整黄金路径仍建议再做一次实操闭环，避免只靠自动化验证带来交付盲区。

## 五、结论

当前项目已经具备可演示、可验证、可继续演进的基础。下一阶段的抓手应聚焦在“交付闭环、体验打磨、生成链路增强预留”三件事上，而不是盲目扩需求。先把现有能力拉通、验证、沉淀，再向下一阶段升级。
