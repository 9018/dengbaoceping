# 项目阶段总结（2026-04-25）

## 1. 当前项目定位

当前仓库已经从“原型化测评工具”进一步推进到“规则驱动的等级保护测评工作台”，并且开始围绕 `md/指导书.md` 中的测评域做能力对齐。

当前系统的主链路已经不再是单点页面或单点接口，而是围绕以下闭环组织：

1. 项目管理
2. 资产与证据管理
3. OCR 识别
4. 字段抽取与人工复核
5. 测评项匹配与模板生成
6. 测评记录复核与审批
7. 项目导出

本轮最重要的推进不是继续堆接口，而是把“字段 -> 测评项 -> 模板 -> 记录”的匹配逻辑，从简单 `required_fields` 命中升级为**可解释、可复核、可人工重选**的评分匹配。

## 3. 与指导书的对齐情况

`md/指导书.md` 当前覆盖了等级保护测评的主要大域，包括：

- 安全物理环境
- 安全通信网络
- 安全区域边界
- 安全计算环境
- 安全管理制度
- 安全管理机构
- 安全管理人员
- 安全建设管理
- 安全运维管理

从当前项目实现情况看，已经开始形成初步覆盖的重点是：

### 3.1 当前已对齐较好的域

#### A. 安全计算环境

当前最成熟的对齐点集中在主机场景：

- 身份鉴别
- 访问控制
- 安全审计
- 入侵防范
- 恶意代码防范

这些域已经进入：

- 字段规则
- 测评项规则
- 模板规则
- 记录生成
- 前端复核展示
- 测试覆盖

#### B. 安全区域边界 / 边界防护

当前已有基础能力：

- 防火墙基础信息
- 安全策略字段抽取
- 边界访问控制记录生成

但颗粒度仍然偏基础，离指导书中“网络区域划分、边界防护、隔离、访问路径控制”等条目还有明显差距。

### 3.2 当前对齐仍然较弱的域

以下域在仓库中仍未形成成熟的规则体系：

- 安全物理环境
- 安全通信网络
- 数据完整性 / 数据保密性 / 数据备份恢复类深度指标
- 安全管理制度
- 安全管理机构
- 安全管理人员
- 安全建设管理
- 安全运维管理

也就是说，当前项目已经从“只有工具能力”进入“开始承接等级保护指标”的阶段，但主要集中在**技术类证据和主机/边界配置类条目**，距离完整覆盖指导书仍有较大空间。

## 6. 当前仍然存在的限制

### 6.1 指导书覆盖仍然偏主机与边界技术域

当前实现最成熟的是：

- 主机安全
- 身份鉴别
- 访问控制
- 安全审计
- 入侵防范
- 恶意代码防范
- 基础边界防护

但对指导书中的大量管理类、物理类、运维类指标仍未形成结构化规则和模板体系。

### 6.2 设备类型识别仍然偏轻量

当前 `MatchingService` 中设备类型判断仍以名称前缀为主：

- `fw* -> firewall`
- `win* / srv* -> server`

这能支撑当前测试样例，但若后续接入更复杂资产类型，规则还需要增强。

### 6.3 边界与网络域模板颗粒度仍不足

指导书中对网络区域、边界隔离、访问路径、自主策略设置等要求更细，而当前模板仍偏“配置摘要型”，还不够贴近正式测评记录措辞。

### 6.4 浏览器级交互验收还未沉淀为固定脚本

当前前端 build 已过，但还未沉淀新的 E2E 自动化，后续页面继续演进时仍有回归风险。

## 7. 阶段复盘

### 7.1 回顾目标

目标是把“字段 -> 测评项 -> 模板 -> 记录”的链路，从简单命中升级为可解释、可复核、可人工干预的评分匹配。

### 7.2 评估结果

目标已经达成，规则、服务、模型、接口、前端、测试六个层面都已同步对齐，没有停留在单点代码修改。

### 7.3 原因分析

这轮工作的关键抓手不是继续堆接口，而是把规则层和记录生成层的结构做扎实：

- 先扩充评价项和模板结构
- 再增强匹配解释
- 再把候选项能力暴露给记录生成和前端
- 最后用测试把闭环固化

### 7.4 可复用规律

后续所有测评域扩展，都不应该再走“先写死模板、再补测试”的散装方式，而应该沿用同一套路：

1. 先从指导书抽测评项
2. 再定义字段规则/模板规则/评分权重
3. 再补前端解释与人工干预入口
4. 最后用 API 回归测试固化
