# 测评项目优化计划（基于指导书，2026-04-25）

## 一、计划目标

在当前“规则驱动测评工作台”已经具备最小可用闭环的基础上，下一阶段的目标不是继续局部补功能，而是**对照 `md/指导书.md` 的测评域，把系统从主机/边界局部能力升级为覆盖更多等级保护测评指标的专业平台**。

当前优化计划的底层逻辑分三层：

1. **先补指标覆盖面**：从指导书中抽象更多测评域，补规则、模板和记录生成能力。
2. **再补交付颗粒度**：让当前记录正文、复核动作、匹配解释更贴近正式测评作业语义。
3. **最后固化质量闭环**：把页面交互、导出、OCR、记录链路沉淀为稳定回归体系。

---

## 二、当前基线判断

### 2.1 当前已经具备的能力

当前系统已经跑通：

- 项目管理
- 资产与证据上传
- Mock / Paddle OCR 边界
- 字段抽取
- 字段人工复核
- 测评项匹配
- 模板生成
- 测评记录生成
- 测评记录复核与审批
- 项目级导出
- 审计留痕

### 2.2 当前已经开始对齐指导书的域

结合 `指导书.md`，当前对齐较好的主要是：

- 安全计算环境
  - 身份鉴别
  - 访问控制
  - 安全审计
  - 入侵防范
  - 恶意代码防范
- 安全区域边界 / 边界防护（基础态）

### 2.3 当前明显未覆盖或覆盖较弱的域

根据指导书，当前还应重点关注：

- 安全物理环境
- 安全通信网络
- 安全区域边界的细粒度隔离要求
- 数据完整性
- 数据保密性
- 数据备份恢复
- 安全管理制度
- 安全管理机构
- 安全管理人员
- 安全建设管理
- 安全运维管理

结论是：

> 当前项目已经具备“技术类证据自动化处理”的主干能力，但距离“按指导书组织完整测评作业”还差一层测评域覆盖和正式化交付语义。

## 三、下一阶段优化方向

## P0：把当前主机/边界评分匹配链路做成真正可交付闭环

### 目标

不是再证明规则能跑，而是证明当前解释型匹配能稳定落地、可在页面真实使用、不会回归。

### 建议动作

1. 补浏览器级回归
   - 记录页查看 Top 3 候选项
   - 人工选择候选项后重新生成
   - 复核页查看缺失字段与匹配原因
   - 审批通过后进入导出中心

2. 固化前端 E2E
   - 覆盖“上传证据 -> OCR -> 抽取 -> 生成记录 -> 人工改选候选项 -> 复核 -> 审批 -> 导出”

3. 补工作区清理规范
   - 明确哪些 markdown 是阶段文档
   - 哪些中间文件不进入最终提交
   - 避免 `pycache` / 临时文件污染分支

### 验收标准

- 页面级真实交互跑通
- 新匹配能力有稳定回归脚本
- 导出链路不受影响

---

## P1：继续扩安全计算环境测评域，做到“主机安全专题成体系”

### 目标

在当前已有的身份鉴别、访问控制、安全审计、入侵防范、恶意代码防范基础上，继续向指导书中的安全计算环境细项靠拢，形成更完整的主机测评规则库。

### 建议扩充方向

1. 身份鉴别
   - 多因素认证
   - 默认口令修改情况
   - 共享账号与唯一身份标识

2. 访问控制
   - 最小权限
   - 管理员分权
   - 特权命令审计
   - 账号禁用/删除时效性

3. 安全审计
   - 审计时间同步
   - 审计保护与防删除
   - 审计集中留存
   - 审计事件范围完整性

4. 入侵防范
   - 高危端口关闭
   - 不必要账号/服务关闭
   - 漏洞修复及时性
   - 基线核查结果结构化

5. 恶意代码防范
   - 实时查杀
   - 病毒库更新时间
   - 定期扫描策略
   - 外来介质检查

### 涉及抓手

- `field_rules.json`
- `evaluation_items.json`
- `templates.json`
- 测试样例 OCR 文本
- 前端记录页解释展示

### 验收标准

- 每个子域至少补齐 2~4 个正式测评项
- 每个测评项有模板、有解释、有测试
- 能生成更接近正式测评口径的记录正文

---

## P2：把安全区域边界 / 安全通信网络做成第二条成熟专题线

### 目标

依据指导书中“网络区域划分、边界防护、隔离、访问路径控制”等要求，补齐网络/边界类规则，使项目不再只偏主机安全。

### 建议扩充方向

1. 网络区域划分
   - VLAN / 安全域划分
   - 管理区 / 业务区 / 测试区隔离

2. 边界防护
   - 防火墙策略规范性
   - 边界访问控制
   - 外联边界最小暴露

3. 技术隔离
   - 重要区域不在边界处
   - 区域间采取可靠隔离手段
   - ACL / 防火墙 / 网闸配置核验

4. 通信安全
   - 通信完整性保护
   - 加密传输配置
   - 网络设备配置一致性

### 当前差距

当前已有：

- 防火墙基础信息
- 安全策略基础字段

当前不足：

- 模板颗粒度仍偏摘要
- 与指导书判断标准未充分对齐
- 缺少拓扑/区域边界语义字段

### 验收标准

- 边界类测评项不再只是基础配置记录
- 记录正文能体现“区域、边界、隔离、策略”这些正式测评语言
- 网络/边界至少形成一套独立专题模板群

---

## P3：补数据安全相关域

### 目标

根据指导书，开始把数据完整性、数据保密性、数据备份恢复等能力纳入规则体系，让记录生成不只停留在设备与配置。

### 建议扩充方向

1. 数据完整性
   - 重要数据传输完整性保护
   - 完整性校验机制

2. 数据保密性
   - 静态敏感数据加密
   - 传输敏感数据加密
   - 密钥管理配置

3. 数据备份恢复
   - 备份策略
   - 恢复演练记录
   - 备份介质与存放要求

### 难点

- 这类字段很多不是单纯配置截图，需要引入更多文档型证据抽取
- 记录模板会更依赖“判断标准 + 预期结果”的表述，而不只是值填空

### 验收标准

- 至少补 1 组数据安全专题模板和测评项
- 能把文档型证据也纳入抽取与生成链路

---

## P4：开始覆盖管理类测评域

### 目标

把系统从“技术测评工作台”逐步推进到“技术 + 管理双线测评工作台”。

### 指导书对应大域

- 安全管理制度
- 安全管理机构
- 安全管理人员
- 安全建设管理
- 安全运维管理

### 建议实现方式

这部分不建议直接复用当前“配置值填空式模板”，而应新增“文档证据型模板”能力：

1. 抽取文档存在性/版本/审批信息
2. 抽取职责、流程、人员、制度要点
3. 生成“已建立/未建立/不完整”的判断型记录正文

### 需要新增的抓手

- 文档型字段规则
- 更适合制度类证据的模板结构
- 记录正文更偏结论型、判断型语义
- 更细的人工复核入口

### 验收标准

- 至少选一个管理类大域做 pilot
- 能从制度/流程文档中抽取核心字段并生成初版测评记录

---

## P5：让记录正文更贴近正式测评交付物语义

### 目标

当前模板已能生成记录，但很多正文仍偏“配置摘要”。下一步要靠近指导书中的“操作步骤、判断标准、预期结果”语言体系。

### 建议动作

1. 模板结构增强
   - 不只保留 `title_template / record_template`
   - 可以考虑在 JSON 里增加：
     - `check_steps_summary`
     - `judgement_basis`
     - `expected_result`

2. 记录正文生成增强
   - 从“罗列字段值”升级为“判断式、结论式、测评式”表述
   - 在缺失字段时，更明确指出缺的是什么、影响哪条判断

3. 复核页增强
   - 显示候选项的测评域信息
   - 显示对应判断依据摘要

### 验收标准

- 记录内容更接近正式测评报告措辞
- 人工复核时能看到更完整的判断依据，不只是字段值

---

## P6：补统一质量闸门

### 目标

防止后续扩规则、扩模板、扩页面时把已有闭环打坏。

### 建议闸门

1. 后端回归

```bash
pytest backend/tests/test_records_api.py -q
pytest backend/tests/services/test_core_services.py -q
pytest backend/tests/test_exports_api.py -q
```

2. 前端构建

```bash
npm --prefix frontend run build
```

3. 浏览器级黄金路径 smoke
   - 生成记录
   - 查看候选项
   - 改选候选项后重生成
   - 审批与导出

4. 迁移验证
   - 涉及模型变更时必须验证 Alembic upgrade/downgrade

---

## 四、推荐执行顺序

建议按下面顺序推进，不要并行摊大饼：

1. **P0**：补浏览器级真实交互回归 + E2E
2. **P1**：继续扩安全计算环境，形成主机专题规则体系
3. **P2**：补安全区域边界 / 安全通信网络专题
4. **P5**：增强记录正文正式化语义
5. **P3**：补数据安全域
6. **P4**：选一个管理类大域做 pilot
7. **P6**：把质量闸门接进持续回归

原因：

- 当前最成熟的抓手是主机与边界技术域，先把这条链打磨深，不要急着横向铺太多域
- 页面真实交互回归是当前最直接的短板，先补闭环再扩能力
- 指导书覆盖扩展应遵循“先技术域、再数据域、最后管理域”的递进顺序

## 五、建议拆解任务

### 任务 1：补记录页 / 复核页真实交互回归

目标：把候选匹配、人工重选、低置信度提示、审批导出串成可重复验证脚本。

### 任务 2：扩安全计算环境测评项与模板

目标：继续把身份鉴别、访问控制、安全审计、入侵防范、恶意代码防范做成更完整专题。

### 任务 3：设计边界/网络专题字段模型

目标：围绕网络区域、边界隔离、访问路径、安全策略细粒度配置补字段与模板。

### 任务 4：增强记录模板语义

目标：让记录正文更接近指导书中的判断标准和预期结果语言。

### 任务 5：规划数据安全 / 管理类大域 pilot

目标：为后续纵向扩域预留方法论，不再局限于设备配置型证据。

## 七、阶段性结论

当前系统已经跨过“从 0 到 1”的原型阶段，下一步真正的竞争力不再是“能不能生成记录”，而是：

1. 能不能按指导书持续扩测评域
2. 能不能把匹配结果解释清楚、让人复核得动
3. 能不能把记录正文打磨到接近正式测评交付物
4. 能不能在持续迭代中保持闭环不回退

换句话说，下一阶段的核心不是再证明系统能开发，而是证明这套平台可以承接越来越多正式测评项目的真实作业要求。
