# 后端接口联调手册

> 更新时间：2026-04-24
> 适用范围：`backend/` 当前最小可用链路
> Base URL：`http://127.0.0.1:8000/api/v1`

---

## 1. 联调主链路

推荐按下面顺序联调：

1. 新建项目
2. 上传证据
3. 执行 OCR
4. 执行字段抽取
5. 查询字段
6. 修正字段 / 复核字段
7. 生成测评记录
8. 修改记录 / 复核记录
9. 查询字段审计日志
10. 查询记录审计日志
11. 发起项目级导出
12. 查询导出任务
13. 下载导出文件

---

## 2. 通用响应格式

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": {},
  "meta": {
    "total": 1
  }
}
```

失败响应：

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "错误说明",
    "details": null
  }
}
```

## 3. 新建项目

### 接口
- `POST /projects`

### 示例请求

```bash
curl -X POST "http://127.0.0.1:8000/api/v1/projects" \
  -H "Content-Type: application/json" \
  -d '{
    "code": "PJT-DEMO-001",
    "name": "等保联调项目",
    "project_type": "等级保护测评",
    "status": "draft",
    "description": "接口联调示例项目"
  }'
```

### 核心字段
- `code`：项目编码，可选，但建议唯一
- `name`：项目名称，必填
- `project_type`：默认 `等级保护测评`
- `status`：默认 `draft`

### 关注返回
- `data.id`：后续所有项目级接口都要使用

## 4. 上传证据

### 接口
- `POST /projects/{project_id}/evidences/upload`

### 示例请求

```bash
curl -X POST "http://127.0.0.1:8000/api/v1/projects/${PROJECT_ID}/evidences/upload" \
  -F "file=@/tmp/firewall_basic.txt" \
  -F "title=防火墙配置截图" \
  -F "evidence_type=screenshot" \
  -F "summary=边界防护设备联调样例" \
  -F "device=服务器A" \
  -F 'tags_json=["demo","export"]' \
  -F "source_ref=manual-demo"
```

### 可选表单字段
- `title`
- `evidence_type`
- `summary`
- `text_content`
- `device`
- `ports_json`
- `evidence_time`：ISO 时间字符串
- `tags_json`：JSON 数组字符串
- `source_ref`
- `category`：默认 `evidence`
- `category_label`：默认 `证据文件`

### 关注返回
- `data.id`：证据 ID，后续 OCR / 抽取 / 记录生成使用
- `data.asset_id`：关联资产 ID

## 5. 执行 OCR

### 接口
- `POST /evidences/{evidence_id}/ocr`

### mock 样例
当前可直接使用的 `sample_id`：
- `firewall_basic`
- `windows_host`
- `windows_host_partial`
- `security_policy`
- `security_policy_missing_action`
- `network_config`
- `noisy_mixed`

### provider 边界
- 默认 `OCR_PROVIDER=mock`，继续走当前 mock OCR 体系
- 当切到真实 provider 时，`sample_id` 可以为空，由真实 provider 自行处理文件
- 当前仓库已预留真实 provider 占位，未接入真实服务前会返回 `REAL_OCR_NOT_CONFIGURED`

### 示例请求

```bash
curl -X POST "http://127.0.0.1:8000/api/v1/evidences/${EVIDENCE_ID}/ocr" \
  -H "Content-Type: application/json" \
  -d '{
    "sample_id": "firewall_basic",
    "force": false
  }'
```

### 关注返回
- `data.ocr_status`
- `data.ocr_provider`
- `data.ocr_result_json`

## 9. 修正字段

### 接口
- `PUT /fields/{field_id}`

### 示例请求

```bash
curl -X PUT "http://127.0.0.1:8000/api/v1/fields/${FIELD_ID}" \
  -H "Content-Type: application/json" \
  -d '{
    "corrected_value": "FW-01-MANUAL",
    "status": "corrected",
    "review_comment": "人工修正设备名称",
    "reviewed_by": "alice"
  }'
```

### 字段状态
- `missing`：抽取未命中，待人工补录/复核
- `extracted`
- `reviewed`
- `corrected`
- `rejected`

### 状态流转约束
- `missing -> reviewed / corrected / rejected`
- `extracted -> reviewed / corrected / rejected`
- `reviewed -> corrected / rejected`
- `corrected -> reviewed / rejected`
- `rejected` 为终态，不允许回退

## 18. 发起项目级导出

### 接口
- `POST /projects/{project_id}/export`

### 示例请求

```bash
curl -X POST "http://127.0.0.1:8000/api/v1/projects/${PROJECT_ID}/export" \
  -H "Content-Type: application/json" \
  -d '{
    "format": "txt"
  }'
```

### 联调前提
- 仅当项目下相关记录状态已到 `approved` / `exported` 时才能发起导出
- 导出成功后，原 `approved` 记录会被更新为 `exported`

### 关注返回
- `data.id`：导出任务 ID
- `data.file_name`
- `data.file_size`
- `data.record_count`
- `data.status`

## 22. 联调注意事项

1. 当前 OCR 为 mock 体系，建议优先使用已存在的 `sample_id`。
2. 字段与记录的状态值有枚举约束，非法值会返回 400。
3. 字段状态从抽取阶段可能直接产出 `missing`，需要人工补录后再继续后续链路。
4. 记录状态必须按 `generated/reviewed -> approved -> exported` 推进，不能跳过审批直接导出。
5. 项目导出当前仅支持 `txt`，传 `pdf` 等格式会返回 `EXPORT_FORMAT_NOT_SUPPORTED`。
6. 审计日志按时间倒序返回，最新操作在前。
7. 项目级导出会把相关 `approved` 记录状态更新为 `exported`；若存在未审批记录会返回 `RECORD_NOT_READY_FOR_EXPORT`。

推荐用下面这组参数快速打通：
- 上传文件：`firewall_basic.txt`
- OCR 样例：`firewall_basic`
- 抽取模板：`security_device_basic`
- 字段修正人：`alice`
- 字段复核人：`bob`
- 记录复核人：`dave`

这条链路最稳定，适合前端、测试先跑第一轮闭环。
