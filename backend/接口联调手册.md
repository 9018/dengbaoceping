## 1. 基础信息

### API Base URL
- 本地直连：`http://127.0.0.1:8000/api/v1`
- 通过前端代理：`/api/v1`

### 通用返回结构

成功：

```json
{
  "success": true,
  "message": "ok",
  "data": {},
  "meta": null
}
```

失败：

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "错误说明",
    "details": null
  }
}
```

## 5. 执行 OCR

### 接口
- `POST /evidences/{evidence_id}/ocr`

### mock 样例
当前可直接使用的 `sample_id`：
- `firewall_basic`
- `windows_host`
- `windows_host_partial`
- `security_policy`
- `security_policy_missing_action`
- `network_config`
- `noisy_mixed`

### provider 边界
- 默认 `OCR_PROVIDER=mock`，继续走当前 mock OCR 体系
- `OCR_PROVIDER=paddle` 时会直接读取已上传文件并执行 PaddleOCR，`sample_id` 可为空且会被忽略
- `OCR_PROVIDER=real` 仍保留为占位 provider，当前会返回 `REAL_OCR_NOT_CONFIGURED`
- Paddle/mock 返回的 OCR 结果统一至少包含：
  - `provider`
  - `status`
  - `full_text`
  - `lines`
  - `processed_at`
- `lines` 中每项结构为：`{ text, confidence, bbox }`
- Paddle 运行期失败会返回 `status=failed` 的结构化结果并持久化到 `ocr_result_json`

### 示例请求

#### mock provider
```bash
curl -X POST "http://127.0.0.1:8000/api/v1/evidences/${EVIDENCE_ID}/ocr" \
  -H "Content-Type: application/json" \
  -d '{
    "sample_id": "firewall_basic",
    "force": false
  }'
```

#### paddle provider
```bash
curl -X POST "http://127.0.0.1:8000/api/v1/evidences/${EVIDENCE_ID}/ocr" \
  -H "Content-Type: application/json" \
  -d '{
    "sample_id": null,
    "force": false
  }'
```

### 关注返回
- `data.ocr_status`
- `data.ocr_provider`
- `data.ocr_result_json`

## 9. 修正字段

### 接口
- `PUT /fields/{field_id}`

### 示例请求

```bash
curl -X PUT "http://127.0.0.1:8000/api/v1/fields/${FIELD_ID}" \
  -H "Content-Type: application/json" \
  -d '{
    "corrected_value": "FW-01-MANUAL",
    "status": "corrected",
    "review_comment": "人工修正设备名称",
    "reviewed_by": "alice"
  }'
```

### 字段状态
- `missing`：抽取未命中，待人工补录/复核
- `extracted`
- `reviewed`
- `corrected`
- `rejected`

### 状态流转约束
- `missing -> reviewed / corrected / rejected`
- `extracted -> reviewed / corrected / rejected`
- `reviewed -> corrected / rejected`
- `corrected -> reviewed / rejected`
- `rejected` 为终态，不允许回退

## 22. 联调注意事项

1. 默认 OCR 为 mock 体系，建议优先使用已存在的 `sample_id`。
2. 当 `OCR_PROVIDER=paddle` 时，`sample_id` 可为空，OCR 结果会统一返回 `lines` 和派生后的 `full_text`。
3. Paddle 运行期异常不会直接打崩接口，而会以 `status=failed` 的结构化结果写入 `ocr_result_json`。
4. 字段与记录的状态值有枚举约束，非法值会返回 400。
5. 字段状态从抽取阶段可能直接产出 `missing`，需要人工补录后再继续后续链路。
6. 记录状态必须按 `generated/reviewed -> approved -> exported` 推进，不能跳过审批直接导出。
7. 项目导出当前仅支持 `txt`，传 `pdf` 等格式会返回 `EXPORT_FORMAT_NOT_SUPPORTED`。
8. 审计日志按时间倒序返回，最新操作在前。
9. 项目级导出会把相关 `approved` 记录状态更新为 `exported`；若存在未审批记录会返回 `RECORD_NOT_READY_FOR_EXPORT`。

## 23. 最小联调样例路径

推荐用下面这组参数快速打通：
- 上传文件：`firewall_basic.txt`
- OCR provider：`mock`
- OCR 样例：`firewall_basic`
- 抽取模板：`security_device_basic`
- 字段修正人：`alice`
- 字段复核人：`bob`
- 记录复核人：`dave`

如果要验证 Paddle 路径，可把 provider 切到 `paddle`，上传真实图片或 PDF 后直接调用 OCR 接口，不再依赖 `sample_id`。
