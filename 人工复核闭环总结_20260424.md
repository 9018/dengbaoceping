# 人工复核闭环阶段总结（2026-04-24）

## 一、阶段目标

本阶段目标是基于现有 `backend/` 能力，补齐“人工复核闭环”最小可用版本，覆盖两条主链路：

1. 字段复核
2. 测评记录复核

同时保持以下约束：

- 不引入复杂审批流
- 先用简单字段落库
- 记录谁改的、什么时候改的、改了什么
- 生成记录时优先使用 `corrected_value`
- 继续保持接口、服务、仓储分层

## 三、本阶段关键修正

### 1. 修正了 records 旧测试与新状态契约冲突

问题本质不是实现不可用，而是旧测试仍然断言历史状态 `needs_review`，与当前人工复核状态契约不一致。

已完成对齐：

- 将 `backend/tests/test_records_api.py` 中对应断言改为 `reviewed`
- 保持 `backend/app/services/record_service.py` 中当前语义不回退

这样做的结果是：

- 旧记录生成测试继续有效
- 新人工复核状态定义不被破坏
- 测试语义与接口契约保持一致

### 2. 修正了字段路由遗漏问题

在复核接口初版实现后，已补上 `fields` 路由注册，避免接口存在但无法访问的链路断裂。

## 五、当前系统状态结论

当前 `backend/` 已经具备以下完整闭环能力：

1. 项目、资产、证据基础 API
2. Mock OCR
3. 字段抽取与规则驱动
4. 条目匹配与模板生成
5. 测评记录落库
6. 字段人工复核
7. 记录人工复核
8. 最小审计留痕
9. 自动化测试与迁移验证

也就是说，最小可用的“上传证据 → OCR → 抽取字段 → 生成记录 → 人工修正字段 → 人工修正记录 → 审计留痕”链路已经跑通。

## 七、阶段复盘

### 1. 回顾目标

目标是先把人工复核闭环做成最小可用版本，并且不破坏此前已完成的记录生成链路。

### 2. 评估结果

结果已经达成，且专项测试、全量测试、迁移验证都已通过。

### 3. 原因分析

唯一明显返工点是状态语义：旧测试仍使用历史状态名，说明“新增能力”与“既有回归基线”在设计上没有第一时间拉通。

### 4. 可复用规律

后续新增状态字段时，必须同步检查三层：

1. 服务层状态语义
2. API schema 对外契约
3. 既有测试断言与历史样例

不然很容易出现“新功能绿、全量回归红”的假闭环。
