# 人工复核闭环后续计划（2026-04-25）

## 一、计划目标

在当前最小可用版本已经闭环的基础上，下一阶段目标不是继续堆功能，而是把“可用”提升为“可联调、可查询、可交付”。

核心抓手分三块：

1. 查询能力补齐
2. 联调与使用手册沉淀
3. 状态与外部能力边界收敛

## 二、当前基线

当前已经完成的能力包括：

- 项目 / 资产 / 证据 API
- Mock OCR 与字段抽取
- PaddleOCR provider 接入（mock 默认、paddle 可切换、real 占位兼容）
- 条目匹配与模板生成
- 测评记录生成
- 字段人工复核
- 记录人工复核
- 最小审计留痕
- Alembic 迁移
- OCR 定向测试通过
- 全量 `backend/tests` 通过

因此，后续计划不再从“无到有”开发，而是从“单点能力”走向“可运营闭环”。

## 三、后续优先级规划

### P1：补齐审计查询能力

#### 目标

让人工复核不是只写入数据库，而是可以被查询、追溯、核查。

#### 建议范围

新增审计查询接口，例如：

- `GET /api/v1/fields/{field_id}/audit-logs`
- `GET /api/v1/records/{record_id}/audit-logs`

#### 建议返回字段

- `id`
- `target_type`
- `target_id`
- `action`
- `changed_fields_json`
- `review_comment`
- `reviewed_by`
- `created_at`

#### 验收标准

- 可以查到字段复核历史
- 可以查到记录复核历史
- 顺序明确，至少按时间倒序
- 有对应 API 测试

#### 风险点

- 如果后续日志量上来，需要补索引和分页
- `changed_fields_json` 展示格式要保证稳定

---

### P2：联调与使用手册沉淀

#### 目标

让后端能力可以被测试、产品、前端直接拿来联调，不依赖读源码。

#### 建议产物

新增一份接口联调文档，建议包含：

1. 新建项目
2. 上传证据
3. 执行 OCR
4. 执行字段抽取
5. 生成测评记录
6. 修正字段
7. 复核字段
8. 修改记录正文
9. 审批记录
10. 查询审计日志（若 P1 已完成）
11. 项目级导出

#### 验收标准

- 每个接口有示例请求
- 每个接口有核心字段说明
- 能直接用于 Postman / curl 调试

#### 风险点

- 如果文档和代码不同步，会快速失效
- 最好在接口稳定后再固化

---

### P3：补齐状态流转约束（已完成）

#### 目标

当前版本允许状态更新，但状态机约束还比较宽松。现已把状态流转规则显式化并收口到服务层。

#### 已落地规则

字段状态：

- `missing -> reviewed / corrected / rejected`
- `extracted -> reviewed / corrected / rejected`
- `reviewed -> corrected / rejected`
- `corrected -> reviewed / rejected`
- `rejected` 终态禁止回退

记录状态：

- `generated -> reviewed`
- `reviewed -> approved`
- `approved -> exported`
- `exported` 终态禁止回退

补充约束：

- 项目导出前，相关记录必须已到 `approved` / `exported`
- 导出成功后，`approved` 记录会被更新为 `exported`

#### 验收结果

- 非法状态跳转会返回明确错误
- 已补充对应 API 测试并通过专项验证

#### 风险点

- 当前仍保持最小状态机，没有引入多人审批和复杂并发控制
- 若后续要支持撤回/退回，需要重新设计状态图

---

### P4：真实 OCR 替换预研（已完成并进入可切换阶段）

#### 目标

在不打破当前 mock 体系的前提下，为未来真实 OCR 接入预留边界。

#### 已落地动作

- 统一 OCR provider 抽象，`OCRService` 继续只依赖 factory
- 保留 `mock_adapter`
- 新增 `paddle_adapter`，并在 `factory.py` 中接入 `OCR_PROVIDER=paddle`
- 保留 `real_adapter` 占位实现与 `OCR_PROVIDER=real` 兼容开关
- OCR adapter 入参统一包含 `filename` 与 `file_path`，为真实文件处理保留边界
- OCR 结果统一补齐：
  - `provider`
  - `status`
  - `full_text`
  - `lines`
  - `processed_at`
- `lines` 每项统一为 `{ text, confidence, bbox }`
- `full_text` 改为由 `lines` 派生
- 运行期 OCR 失败统一转换为结构化失败 payload
- Paddle 引擎改为 lazy init / 单例缓存
- 保持默认 `OCR_PROVIDER=mock`，现有专项测试和全量测试不受影响

#### 当前结论

- 真实 OCR 的最小替换点已经从“预留边界”推进到“代码已可切换”：`app/services/ocr/factory.py`
- 当前 mock / paddle / real 三态分工已经明确：
  - `mock`：默认开发与 CI
  - `paddle`：真实 OCR provider
  - `real`：历史占位兼容
- 当前阻塞点不再是 service 层结构，而是 Paddle 真实依赖安装、真实图片/PDF smoke 与运行环境差异
- OCR 结果结构已经统一，后续 provider 扩展可以继续复用同一结果契约

#### 后续建议动作

1. 在真实环境安装 Paddle 依赖并执行一次图片 / PDF smoke
2. 补 `PADDLE_OCR_*` 配置的运行环境说明
3. 如需扩真实云 OCR，再在 `real_adapter` 中实现鉴权/轮询 provider
4. 持续把新增 provider 对齐到统一 `ocr_result_json` 结构

## 四、推荐执行顺序

建议按下面顺序推进，而不是并行摊大饼：

1. 真实 PaddleOCR 环境 smoke
2. 审计查询接口
3. 联调文档持续对齐
4. 状态流转与导出链路细化
5. 真实云 OCR provider 预研（如业务需要）

原因很简单：

- 项目级导出、状态流转、接口联调文档、PaddleOCR 代码接入都已经完成主体闭环
- 当前重点应从“有没有接好 provider”切到“真实环境能否稳定跑通”
- 在真实 Paddle 路径跑通之后，再决定是否有必要引入更重的云 OCR provider

## 六、验收口径

后续每个阶段都建议继续沿用同一套闭环标准：

### 1. 代码验证

```bash
python3 -m compileall backend/app backend/tests
backend/.venv/bin/python -m pytest backend/tests -q
```

### 2. 迁移验证

如涉及表结构调整，执行：

```bash
DATABASE_URL=sqlite:////tmp/cpgj_next.db backend/.venv/bin/alembic -c backend/alembic.ini upgrade head
DATABASE_URL=sqlite:////tmp/cpgj_next.db backend/.venv/bin/alembic -c backend/alembic.ini downgrade base
```

### 3. 链路验证

至少验证：

1. 上传证据
2. OCR
3. 字段抽取
4. 生成记录
5. 字段复核
6. 记录复核
7. 审计查询 / 导出（如果该阶段已实现）

## 七、实施原则

后续继续遵循三条原则：

1. **先补闭环，再补花活**
   - 先把查询、导出、联调做扎实，不急着上复杂审批流。

2. **先保证契约一致，再扩功能**
   - 每次新增状态、字段、接口时，同步对齐 schema、service、test。

3. **先最小可用，再抽象复用**
   - 不做提前设计，不为了未来想象引入复杂结构。

## 八、阶段性结论

当前系统已经跨过“从 0 到 1”的门槛，后续工作的底层逻辑已经不是“还能不能做出来”，而是“怎么把它打磨成别人能稳定使用的能力”。

换句话说，下一阶段的核心不是再证明能开发，而是证明这套链路能长期闭环。
