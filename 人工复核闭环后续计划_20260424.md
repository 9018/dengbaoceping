# 人工复核闭环后续计划（2026-04-24）

## 一、计划目标

在当前最小可用版本已经闭环的基础上，下一阶段目标不是继续堆功能，而是把“可用”提升为“可联调、可查询、可交付”。

核心抓手分三块：

1. 查询能力补齐
2. 导出链路落地
3. 联调与使用手册沉淀

## 二、当前基线

当前已经完成的能力包括：

- 项目 / 资产 / 证据 API
- Mock OCR 与字段抽取
- 条目匹配与模板生成
- 测评记录生成
- 字段人工复核
- 记录人工复核
- 最小审计留痕
- Alembic 迁移
- 全量 `backend/tests` 通过

因此，后续计划不再从“无到有”开发，而是从“单点能力”走向“可运营闭环”。

## 三、后续优先级规划

### P1：补齐审计查询能力

#### 目标

让人工复核不是只写入数据库，而是可以被查询、追溯、核查。

#### 建议范围

新增审计查询接口，例如：

- `GET /api/v1/fields/{field_id}/audit-logs`
- `GET /api/v1/records/{record_id}/audit-logs`

#### 建议返回字段

- `id`
- `target_type`
- `target_id`
- `action`
- `changed_fields_json`
- `review_comment`
- `reviewed_by`
- `created_at`

#### 验收标准

- 可以查到字段复核历史
- 可以查到记录复核历史
- 顺序明确，至少按时间倒序
- 有对应 API 测试

#### 风险点

- 如果后续日志量上来，需要补索引和分页
- `changed_fields_json` 展示格式要保证稳定

---

### P2：落地记录导出链路

#### 目标

把当前状态枚举中的 `exported` 从占位状态变成真实业务动作。

#### 建议范围

新增：

- `POST /api/v1/records/{record_id}/export`

建议导出方向：

- 先导出为文本或 markdown
- 落盘到 `backend/exports/`
- 导出成功后更新：
  - `status = exported`
  - 写入审计日志

#### 验收标准

- 导出文件实际落地
- 状态从 `approved/reviewed` 更新到 `exported`
- 导出接口有自动化测试
- 失败场景有明确报错

#### 风险点

- 需要先确认导出格式是否只是内部文本，还是未来要转 Word/PDF
- 如果后续要求模板化导出，结构设计要留扩展口

---

### P3：补一版人工联调手册

#### 目标

让后端能力可以被测试、产品、前端直接拿来联调，不依赖读源码。

#### 建议产物

新增一份接口联调文档，建议包含：

1. 新建项目
2. 上传证据
3. 执行 OCR
4. 执行字段抽取
5. 生成测评记录
6. 修正字段
7. 复核字段
8. 修改记录正文
9. 审批记录
10. 查询审计日志（若 P1 已完成）
11. 导出记录（若 P2 已完成）

#### 验收标准

- 每个接口有示例请求
- 每个接口有核心字段说明
- 能直接用于 Postman / curl 调试

#### 风险点

- 如果文档和代码不同步，会快速失效
- 最好在接口稳定后再固化

---

### P4：补齐状态流转约束

#### 目标

当前版本允许状态更新，但状态机约束还比较宽松。下一阶段可考虑把状态流转规则显式化。

#### 建议规则

字段状态可考虑：

- `extracted -> reviewed`
- `extracted -> corrected`
- `extracted -> rejected`
- `reviewed -> corrected`

记录状态可考虑：

- `generated -> reviewed`
- `reviewed -> approved`
- `approved -> exported`

#### 验收标准

- 非法状态跳转会返回明确错误
- 有对应单元测试

#### 风险点

- 过早做复杂状态机会抬高当前版本复杂度
- 建议放在 P1/P2 之后做

---

### P5：真实 OCR 替换预研

#### 目标

在不打破当前 mock 体系的前提下，为未来真实 OCR 接入预留边界。

#### 建议动作

- 统一 OCR provider 抽象
- 保留 `mock_adapter`
- 增加真实 provider 接口占位
- 明确超时、失败重试、图片/PDF 输入差异

#### 验收标准

- mock 仍可运行
- 新 provider 不影响现有测试
- 有最小设计说明

#### 风险点

- 外部 OCR 依赖会引入网络、额度、鉴权、稳定性问题
- 没有真实需求前不建议过度实现

## 四、推荐执行顺序

建议按下面顺序推进，而不是并行摊大饼：

1. 审计查询接口
2. 导出接口
3. 联调文档
4. 状态流转约束
5. 真实 OCR 预研

原因很简单：

- P1、P2 是把现有闭环补成真正可用链路
- P3 是把可用链路交付给别人使用
- P4、P5 属于增强项，价值高但优先级略低

## 六、验收口径

后续每个阶段都建议继续沿用同一套闭环标准：

### 1. 代码验证

```bash
python3 -m compileall backend/app backend/tests
backend/.venv/bin/python -m pytest backend/tests -q
```

### 2. 迁移验证

如涉及表结构调整，执行：

```bash
DATABASE_URL=sqlite:////tmp/cpgj_next.db backend/.venv/bin/alembic -c backend/alembic.ini upgrade head
DATABASE_URL=sqlite:////tmp/cpgj_next.db backend/.venv/bin/alembic -c backend/alembic.ini downgrade base
```

### 3. 链路验证

至少验证：

1. 上传证据
2. OCR
3. 字段抽取
4. 生成记录
5. 字段复核
6. 记录复核
7. 审计查询 / 导出（如果该阶段已实现）

## 七、实施原则

后续继续遵循三条原则：

1. **先补闭环，再补花活**
   - 先把查询、导出、联调做扎实，不急着上复杂审批流。

2. **先保证契约一致，再扩功能**
   - 每次新增状态、字段、接口时，同步对齐 schema、service、test。

3. **先最小可用，再抽象复用**
   - 不做提前设计，不为了未来想象引入复杂结构。

## 八、阶段性结论

当前系统已经跨过“从 0 到 1”的门槛，后续工作的底层逻辑已经不是“还能不能做出来”，而是“怎么把它打磨成别人能稳定使用的能力”。

换句话说，下一阶段的核心不是再证明能开发，而是证明这套链路能长期闭环。
